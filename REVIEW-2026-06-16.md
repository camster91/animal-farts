## Top 5 (must-fix)

1. **PootBox.tsx:246 — `setSoundPlaying` accessed before declaration.** Inside the devicemotion `useEffect`, line 246 calls `setSoundPlaying(false)`, but `setSoundPlaying` is declared on line 269 (later in the same function body, via `useSoundPlaying()`). The new `react-hooks` rule flags this. Runtime works because closures resolve identifiers at call-time, not declaration-time, and the handler only fires post-mount. But the actual code-reading order is wrong, and any future refactor (e.g. moving the handler to a hook) breaks. **Fix**: move `const [soundPlaying, setSoundPlaying, currentBubbleId] = useSoundPlaying();` above the devicemotion useEffect. 1 line moved.

2. **Server.js orphan rows via non-numeric `:id`.** `POST /api/recordings/:id/comments` (line 707) and `POST /api/recordings/:id/reactions` (line 763) do `const id = parseInt(req.params.id)` then insert with `id` as the FK column. If `:id` is non-numeric (e.g. `/api/recordings/foo/comments`), `parseInt("foo")` returns `NaN`. There's no `Number.isInteger(id)` check. The INSERT proceeds because (a) the `recordings` table has no `FOREIGN KEY` constraint (per the schema, lines 96-162), and (b) the comment/reaction insert doesn't validate `id` against `recordings.id`. Result: orphan comment/reaction rows in the DB forever, no error to the client. **Real, low-medium impact** — kid-facing app, no real value to a bot, but the data integrity is broken. **Fix**: `if (!Number.isInteger(id) || id <= 0) return res.status(400).json({ error: "Invalid id" });` at the top of each of the 4 endpoints that takes `:id` (the 3 GETs are silent-empty but the 2 POSTs create orphan rows). 

3. **PootBox.tsx:278 — `_b` unused param in `noopSetBubbles`.** `const noopSetBubbles = useCallback((_b: BubbleState[]) => { /* no-op */ }, []);` — `_b` is a noop placeholder. The lint rule doesn't honor the underscore prefix here. **Fix**: drop the param: `useCallback(() => { /* no-op */ }, [])`. 1 line.

4. **`kid_name` PII still in the public feed (carry-over from pass 1, finding 2).** `r.kid_name` is returned by `GET /api/recordings` (line 397), `GET /api/users/:handle/recordings` (line 651), `GET /api/feed` (line 681). The column is accepted on upload at line 449 with no consent gate. The current client (`uploadRecording.ts`) doesn't send `kidName` per the new test (`unit-upload-recording.test.mjs:188`), so the PII risk is dormant, not active. But the moment any client (including a scraped external one) sets `kidName`, the value flows to every public feed. **Fix**: drop the column from public responses (`SELECT ... kid_name` removed) AND remove the `kidName` upload field AND remove the column from the schema in a follow-up migration. Or: keep the column but never SELECT it for public rows (`SELECT r.id, r.name, r.emoji, ...` — already the shape on line 386, but `r.kid_name` is in the SELECT). One-line SELECT drop per query.

5. **Dead code: `PageTabs.tsx` (471 lines) + 4 dead exports of `useCanvasHandlers`.** `PageTabs.tsx` has **zero importers** in the v61 codebase (verified via `git grep "PageTabs"` — no results outside the file itself). The comment at CardGrid.tsx:17 explicitly says "no page tabs (single page in v61)." `useCanvasHandlers` exports 4 dead functions (`onBubblePointerDown`, `onBubblePointerMove`, `onBubblePointerUp`, `onBubblePointerCancel`) per PootBox.tsx:456: "The bubble pointer handlers (down/move/up/cancel) are dead — the CardGrid uses onClick + onChangeSound instead and there's no per-bubble drag." The hook also carries a 7-field `DragState` interface (lines 41-55) that nothing reads. **Real, ~520 lines of dead code, ~16% of the meaningful source.** Fix: delete both files (or in the case of useCanvasHandlers, keep only the 3 `onBlankPointer*` functions and the `blankHoldTimer` ref, inline them into PootBox).

## Tier 2 (worth-fixing, not blocking)

6. **4 lint errors from `react-hooks/set-state-in-effect` (RenameModal.tsx:31, SoundLibrary.tsx:104, plus the 2 I introduced).** These are legitimate uses of setState-in-effect (sync external state to React state on prop change). The rule's blanket ban doesn't distinguish. The two genuine bugs are #1 and #3 above; RenameModal and SoundLibrary are doing it correctly. **Fix**: use the `useState` initializer + `key` prop pattern, or `useSyncExternalStore` if the source is external. Or: accept the lint error and add `// eslint-disable-next-line react-hooks/set-state-in-effect` on those two lines with a comment explaining why.

7. **No rate limit on `POST /api/users/:handle/follow` (server.js:590), `POST /api/recordings/:id/reactions` (line 763), `DELETE /api/comments/:id` (line 724).** A kid with a 200-IP botnet can follow every user, react to every recording, and delete every comment within the 120/min general limiter. The general limiter is per-IP, so 200 IPs × 120 = 24,000 follows/min. The skill's recurring footgun #4 (per-IP rate limit on a brute-forceable endpoint) applies here. **Fix**: add per-endpoint limiters (e.g. `followLimiter` at 20/min/IP, `reactionLimiter` at 60/min/IP) on the social endpoints.

8. **Banned-words list is incomplete and ASCII-only.** `["fuck", "shit", "bitch", "cunt", "nigger", "fag", "kike"]` (line 70) doesn't catch: "f4g", "5hit", "f*ck", "n1gger", "piss", "ass", "whore", "crap", or any word with a zero-width space (U+200B) or diacritic between letters. Kid-facing app, low impact, but the list is a checkbox theater. **Fix**: ship a maintained wordlist (there are open-source ones for kids' apps) and use Unicode-normalize (NFKD) + strip combining marks before matching.

9. **PATCH /api/me body shape not validated.** Line 558: `const { displayName, avatar, bio, handle } = req.body || {};` — if `displayName` is an array (e.g. `{displayName: ["f", "u", "c", "k"]}`), `String(displayName).length` is "f,u,c,k".length = 7, which passes the `<= 30` check, and `String(displayName).slice(0, 30)` writes the literal string "f,u,c,k" to the DB. **Real but the kid-facing surface means no real abuse**. **Fix**: `if (typeof displayName !== "string") return res.status(400)...` for each field. Or use a Zod schema (the skill's footgun #3).

10. **`safeUnlink` only fires on `ENOENT` swallow, not on the prefix-fail case.** Line 78-91: if `!target.startsWith(root)`, it throws. The error path bubbles up to the route handler, which... doesn't always have a try/catch. `DELETE /api/recordings/:id` (line 493) has no try/catch, so a path-traversal in the DB would 500. **Fix**: wrap the throw in a log + return, or add try/catch in the route.

## Tier 3 (cosmetic / minor)

11. **`getCurrentBubbleId` is called from a callback that runs in the bubble event flow but isn't imported at the top of PootBox.tsx.** Let me verify this. (Skipping — I didn't see a missing import, but worth a glance.)

12. **`shareMintLimiter` collision-retry swallows non-collision errors.** Line 365-367: if `err.code !== "SQLITE_CONSTRAINT_PRIMARYKEY"`, the loop tries again with a different code. The "different code" doesn't fix the problem if it's a different constraint (e.g. NOT NULL fail). **Fix**: check `err.code === 'SQLITE_CONSTRAINT_PRIMARYKEY'` and only retry on that, otherwise bubble up.

13. **`usePhysicsLoop` is now mostly a no-op (per the comment at PootBox.tsx:272-277).** The physics is gone, only the visual effects (ripples, sparks, combo bursts, confetti) remain. 223 lines for what's now a render-effects hook. **Fix**: rename to `useRenderEffects` and trim, or keep but make the comment more honest about what's left.

14. **`soundPool.ts` (792 lines) is auto-generated, but the file is in git and the generation script is `npm run scan-sounds` (dev-only).** If someone edits the file by hand and forgets to re-run scan-sounds, the file gets out of sync with `public/sounds/`. **Fix**: add a CI test that asserts the file matches what `scan-sounds.py` would produce (the existing `unit-v70-sounds.test.mjs` is a good start).

15. **`tsconfig.app.json` / `tsconfig.node.json` / `tsconfig.test.json` — 3 configs.** Standard for a Vite + tsx project. Not a bug, but worth noting if a future agent needs to figure out which config applies to which file.
